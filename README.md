### **Category Two: Algorithms and Data Structures**

#### **Artifact: CS-340 Grazioso Salvare Animal Rescue Dashboard**

**Current Limitations:**

1. **Linear Search Operations:** All filtering and searching operations iterate through the entire dataset sequentially, resulting in O(n) time complexity
2. **No Data Structure Optimization:** Uses basic Python lists and dictionaries without leveraging advanced data structures
3. **Inefficient Sorting:** Uses Python's built-in sort without implementing custom algorithms optimized for specific use cases
4. **No Caching Mechanism:** Repeated queries fetch from database each time with no intelligent caching

**Why This Artifact:**

- **Real-World Dataset:** 10,000+ animal records make performance improvements measurable and meaningful
- **Multiple Search Criteria:** Searching by ID, breed, age, rescue type, and location provides opportunities to implement different optimized data structures
- **Quantifiable Performance:** Can measure and document time complexity improvements (O(n) ‚Üí O(log n) or O(1))

**Planned Enhancements:**

**1. Binary Search Tree (BST) for Animal ID Lookup**
- **Purpose:** Efficient O(log n) average-case search time vs. current O(n) linear search
- **Use Case:** Quick retrieval when user clicks on animal in table
- **Improvement:** ~10x faster searches

**2. Hash Table for Breed Lookup**
- **Purpose:** O(1) average-case lookup for breed filtering
- **Use Case:** Instant breed filtering (e.g., "Show all Labrador Retrievers")
- **Improvement:** ~35x faster breed queries

**3. Merge Sort for Multi-Field Sorting**
- **Purpose:** Efficient O(n log n) sorting with guaranteed performance and stability
- **Use Case:** Fast sorting of 10,000+ records by any column
- **Improvement:** 30% faster + stability for tie-breaking

**4. QuickSelect for Finding Top N Animals**
- **Purpose:** Find top N animals without full sort - O(n) average vs. O(n log n)
- **Use Case:** "Show me the 10 youngest dogs suitable for water rescue"
- **Improvement:** ~5x faster for top-N queries

**Skills Demonstrated:**

- **Algorithm Implementation:** BST recursive/iterative operations, hash function design with collision handling, divide-and-conquer (merge sort), partitioning algorithms (quickselect)
- **Complexity Analysis:** Big O notation for all operations, best/average/worst case analysis, time vs. space trade-offs
- **Data Structure Selection:** BST for ordered data with range queries, hash table for categorical lookups, merge sort for stable sorting, QuickSelect for top-N problems
- **Professional Practices:** Algorithm documentation, unit testing (edge cases, performance), code optimization

**Course Outcomes Alignment:**

**Outcome 3: Algorithmic Principles**
- Design solutions using classical algorithms (BST, hash tables, merge sort)
- Analyze time/space complexity with Big O
- Manage trade-offs (BST O(log n) vs Hash O(1), stable vs unstable sort)
- Compare alternatives with justification

**Outcome 2: Professional Communication**
- Pseudocode with complexity annotations
- Performance benchmarking with clear metrics

**Outcome 4: Well-Founded Techniques**
- Classical algorithms (von Neumann merge sort, Hoare quickselect)
- Measurable value: 5-35x performance improvements

### **Category Three: Databases**

#### **Artifact: CS-360 Weight Tracker Android Application**

**Origin:** CS 360 - Mobile Architecture and Programming, August 2024

**Current State:**
- Android app using SQLite local database
- Three tables: Users, WeightEntries, Goals
- All data stored on device only

**Why Enhance:**

Migrate to MongoDB to demonstrate NoSQL database design and modern database practices.

**Planned Enhancements:**

**Migration: SQLite ‚Üí MongoDB**

**Core Changes:**
1. Replace SQLite tables with MongoDB collections
2. Redesign schema using NoSQL best practices (embedded documents vs. references)
3. Create indexes for query optimization
4. Implement aggregation pipelines for analytics
5. Add REST API (simple Express.js) to connect Android app to MongoDB

**Schema Transformation:**

**Before (SQLite - Relational):**
```
Users: id, username, password
WeightEntries: id, user_id, weight, date
Goals: id, user_id, target_weight, target_date
```

**After (MongoDB - Document-Based):**
```javascript
// users collection
{
  _id: ObjectId,
  username: String (indexed),
  email: String,
  passwordHash: String,
  profile: {
    targetWeight: Number,
    height: Number,
    unit: "lbs"
  }
}

// weightEntries collection
{
  _id: ObjectId,
  userId: ObjectId (indexed),
  weight: Number,
  date: Date (indexed),
  note: String
}

// goals collection
{
  _id: ObjectId,
  userId: ObjectId (indexed),
  targetWeight: Number,
  startWeight: Number,
  targetDate: Date,
  status: "active" | "achieved",
  progressHistory: [{
    date: Date,
    weight: Number
  }]
}
```

**Key Improvements:**
- Embedded documents (profile inside user)
- Proper Date objects instead of TEXT strings
- Arrays for progress history (MongoDB native)
- ObjectId for relationships instead of integers

**Skills Demonstrated:**

- **NoSQL Database Design:** Document-based schema vs. relational tables, embedded documents (denormalization), reference patterns (one-to-many), data type optimization
- **MongoDB Expertise:** CRUD operations with MongoDB Java Driver, aggregation pipeline syntax, index creation and optimization, query performance analysis
- **Database Migration:** ETL process, data type conversion, foreign key ‚Üí ObjectId reference mapping, data validation post-migration
- **Performance Optimization:** Single-field indexes for unique constraints, compound indexes for multi-field queries, index selection strategy

**Course Outcomes Alignment:**

**Outcome 4: Well-Founded Techniques**
- MongoDB is industry-standard NoSQL database
- Demonstrates modern document-based design
- Aggregation pipelines are MongoDB best practice
- Local MongoDB useful for development/testing environments

**Outcome 5: Security**
- Password hashing (bcrypt) instead of plain text
- Input validation for database operations
- NoSQL injection prevention (parameterized queries)

**Outcome 2: Professional Communication**
- Schema diagrams comparing SQL vs NoSQL
- Migration documentation
- Query comparison examples
- Index justification
